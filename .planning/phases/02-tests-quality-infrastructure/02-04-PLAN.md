**Canary test (cross-extension.test.ts)** — distinct shape:

```typescript
/**
 * Cross-extension canary — two extensions, one session, PANTRY_KEYS round-trip.
 *
 * Asserts the jiti-isolation assumption (PITFALLS §2) holds: when
 * createTestSession loads both a publisher and a consumer in the same session,
 * the consumer can read the publisher's API via getGlobal(PANTRY_KEYS.<key>).
 *
 * Pair: dragon-parchment (publishes pantry.parchment at dragon-parchment.ts:1873)
 *       + dragon-tongue  (consumes  pantry.parchment at dragon-tongue.ts:42)
 *
 * If this fails, two extensions load correctly in isolation but the
 * globalThis-symbol bridge between them is broken — which is the specific
 * regression this canary exists to catch.
 *
 * No direct ../extensions/ imports. No behavioral tool-call smokes.
 */
import { describe, it, expect } from "vitest";
import { createTestSession } from "../helpers/createPiTestSession.ts";
import { PANTRY_KEYS, getGlobal } from "../../lib/globals.ts";

describe("cross-extension: dragon-parchment ↔ dragon-tongue (pantry.parchment round-trip)", () => {
  it("publisher exposes pantry.parchment and consumer can read it", async () => {
    const session = await createTestSession({
      extensions: ["dragon-parchment", "dragon-tongue"],
    });

    // Publisher side — from the jiti-loaded extension runtime:
    const parchmentApi = getGlobal(PANTRY_KEYS.parchment);
    expect(parchmentApi).toBeDefined();

    // (Consumer-side assertion — if the harness exposes dragon-tongue's runtime
    // state, assert the consumer actually resolved the parchment API. If not,
    // the "published and reachable via getGlobal" assertion above is the canary
    // — it IS the cross-extension round-trip because both extensions ran in the
    // same jiti context. This is the minimum meaningful assertion per D-05.)

    await session.close?.();
  });
});
```

**Extension loading order:** inside `createTestSession({ extensions: ["dragon-parchment", "dragon-tongue"] })`, dragon-parchment must load first so its `registerGlobal(PANTRY_KEYS.parchment, api)` runs before dragon-tongue's `getGlobal(PANTRY_KEYS.parchment)`. If the harness does not guarantee left-to-right load order, the test either (a) asserts only the publisher side (`getGlobal` of the published key) which is order-agnostic, or (b) documents the ordering assumption in the top-of-file comment and relies on alphabetical or declaration order. Per Task 2 of this plan, the executor verifies which case applies before committing.
