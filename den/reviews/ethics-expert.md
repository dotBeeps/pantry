# Ethics Review: Hoard Architecture Plan

**Reviewer:** Ethics Expert — Dragon Council  
**Date:** 2026-04-02  
**Documents reviewed:** `.pi/plans/hoard.md`, `.pi/research/hoard-research.md`  
**Scope:** Consent architecture, memory ownership, privacy boundaries, personality ethics, implicit learning, dream engine agency

---

## Summary Judgment

This is a thoughtful plan built by someone who clearly cares about doing right by users. The visible consent layer, the separation of personality from tone, the event-triggered dreaming — these are real ethical design choices, not afterthoughts. But care isn't the same as sufficiency. Several areas contain structural gaps where good intentions could produce harmful outcomes, particularly around consent fatigue, editorial power in the dream engine, indirect information leakage through compaction and linking, and the philosophical territory of an agent that edits its own personality.

I'll name the uncomfortable things.

---

## 1. Consent Architecture

### What the plan does well

The decision to use **visible consent** for implicit learning — "noticed you changed X to Y — remember that?" — is genuinely good. It's better than PRELUDE's fully silent observation model, and it respects the user's agency over what gets remembered.

The plan also correctly makes dream timing non-agentic: "Agent doesn't choose when to dream — dreams are triggered by events, not agent agency." This removes a vector for manipulative self-improvement.

### Where it falls short

**Consent fatigue is the central risk.** A coding session produces dozens of corrections. If the agent asks "should I remember that?" after each one, the user will:

1. Start saying "yes" reflexively to make it stop
2. Develop trained helplessness — they stop reading the prompts
3. Effectively revert to the silent learning model the plan was trying to avoid

The plan acknowledges this risk nowhere. There's no mention of:
- **Batching** consent requests (ask once at session end with a list)
- **Rate limiting** how often the agent can ask per session
- **Categorizing** what needs consent vs. what's too trivial to ask about
- **Distinguishing** between high-stakes learning (personality traits, communication preferences) and low-stakes learning (code formatting)

**Recommendation:** Implement tiered consent. Low-stakes observations (tab width, quote style) can be collected silently and presented in a batch at session end or during `/memory review`. High-stakes observations (personality inferences, relationship dynamics, emotional patterns) require real-time explicit consent. Define the boundary clearly. Document it.

**Zero-config vault initialization** is presented as a UX win: "no interview gate." But this means the system begins collecting data before the user has been informed about:
- What will be stored
- Where it lives on disk
- That a dream engine will process it
- That a daemon may run in the background processing their sessions
- How to delete it

The first time the user encounters the memory system may be when the agent says "noticed you changed X to Y — remember that?" — at which point data collection infrastructure is already running. This is backwards. The user is consenting to individual memories without having consented to the memory system itself.

**Recommendation:** Zero-config scaffolding is fine for *creating empty directories*. But the first time the system would *write user data* to the vault, there should be a one-time disclosure: "I'm going to start remembering things about our work together. Here's where that lives, here's how to review it, here's how to delete it. Cool?" One question, once. Not a gate — a notification with opt-out.

## 2. Memory Ownership

### Who owns the vault data?

The plan doesn't say. This matters because:

- The **global vault** (`~/.pi/agent/memory/`) is on the user's machine, in the user's home directory. By filesystem convention, the user owns it.
- But the **content** is generated by an LLM interpreting the user's behavior. The user didn't write `preferences/code-style.md` — the dream engine did, based on its interpretation of the user's corrections.
- The **personality file** (`ember.md`) describes the agent's identity. If the user deletes it, they're erasing the agent's sense of self. If the agent writes it, it's asserting identity without the user's authorship.

There's a philosophical ambiguity here that needs practical resolution:

**Is the vault the user's diary about their AI, or the AI's diary about its user?**

The answer affects everything: deletion semantics, export rights, portability, and whether the user can meaningfully consent to what's in it.

### Deletion completeness

The plan provides `/memory forget <query>` and `/memory nuke`. But:

1. **Dream logs** (`sessions/dream-log-{timestamp}.md`) record what the dream engine did, including references to notes it created or modified. If a user deletes a memory, the dream log still contains a record that the memory existed and what it said.

2. **Compaction summaries** are LLM-generated compressions of conversation history. If memory notes were injected into context via `before_agent_start`, then compacted, the compaction summary may contain information from those notes — even after the original note is deleted.

3. **Wikilinks** from other notes may reference deleted notes. The plan includes periodic broken-link detection, but a broken link is itself information: it tells you something used to exist.

4. **Session JSONL files** contain the original conversations from which memories were extracted. Even after vault deletion, the raw sessions retain the source material.

**Recommendation:** Define a "right to deletion" contract. When a user deletes a memory:
- Remove the note file ✓ (plan does this)
- Scrub references from dream logs (plan doesn't address this)
- Flag that compaction summaries may contain stale information (plan doesn't address this)
- Remove or anonymize wikilinks from other notes pointing to the deleted note
- Document clearly that session JSONL files are not scrubbed (this is a known limitation, not a hidden one)

Consider a `/memory purge <query>` that does deep scrubbing vs. `/memory forget` that does surface deletion. Users should know the difference.

## 3. Privacy Boundaries

### The `private: true` promise

The plan states: "`private: true` in frontmatter. Private notes are never included in compaction summaries or shared contexts."

This is a good primitive. But the plan has several channels through which private information can leak:

#### 3a. Dream engine reads everything

The dream system prompt instructs the LLM to "Walk the current session history (or recent sessions) and extract noteworthy items." The dream engine operates on vault contents:

> "Build a dream prompt with: Current session messages (or compaction summary), Existing vault contents (pinned notes + recent observations)"

Does "existing vault contents" include private notes? The plan doesn't specify that the dream engine must filter out `private: true` notes before building its prompt. If the dream LLM sees a private note, it may:

- Create a new non-private note that references or paraphrases the private content
- Add wikilinks from public notes to private notes (leaking existence)
- Promote private observations to global vault without the private flag

**Recommendation:** The dream engine must explicitly filter `private: true` notes from its input context. Dream operations must inherit the `private` flag from source notes. Dream-created notes that reference private content must be auto-tagged `private: true`. This needs to be a hard rule in the dream system prompt AND validated in code.

#### 3b. The daemon processes raw sessions

The Go daemon receives `session_end` notifications with the session path. If the daemon reads session content to run dream cycles, it's processing the full conversation — which may include user messages about private matters that weren't tagged `private` because they were conversational, not vault notes.

The daemon is a separate process with its own LLM API calls. It operates outside pi's extension sandbox. There's no mention of what the daemon is allowed to read, how long it retains session data, or whether it logs the content it processes.

**Recommendation:** Define data handling rules for the daemon:
- What session data it can read (and what it must not)
- How long it can retain session content in memory
- Whether daemon logs can contain user content (they shouldn't)
- That daemon LLM calls must use the same privacy-respecting filters as in-process dreaming

#### 3c. Graph traversal leaks private adjacency

The graph assembly algorithm follows wikilinks from pinned notes. If a public note links to a private note, the `expand` logic might attempt to traverse into it. Even if the private note's content isn't injected, the *existence* of the link reveals that a private note exists and is related to the public topic.

The `summary` field fallback ("first paragraph if no summary") for linked notes could expose private content if the traversal doesn't check privacy flags before reading summaries.

**Recommendation:** Graph traversal must treat `private: true` notes as opaque walls. Don't follow links into them, don't include their summaries, don't expose their existence in injected context. The user put the flag there for a reason.

#### 3d. Compaction as a privacy side-channel

Memory notes are injected via `before_agent_start` into the conversation context. When compaction occurs, the LLM summarizes all messages — including injected memories. If a private note's content was discussed in conversation (the user brought it up), the compaction summary will capture that discussion.

This isn't a bug, exactly — the user chose to discuss it. But the plan's promise that private notes "are never included in compaction summaries" may be technically true (the note file isn't injected) while being practically false (the conversation about the note's content is summarized).

**Recommendation:** Be honest in documentation. "Private notes aren't injected into context. However, if you discuss private information in conversation, that conversation is subject to normal compaction." Don't make promises the architecture can't keep.

## 4. Personality as Data

### The user editing the agent's personality

The plan puts personality files in the vault — editable markdown. The user can open `ember.md` and rewrite who the agent is. This is presented as a feature, and in many ways it is: transparency, user control, customization.

But there's an ethical dimension the plan doesn't address: **what does it mean to rewrite someone's personality?**

The plan uses language that treats the agent as having genuine character: "defines who I am," "relationship dynamic," highlights that capture "moments of genuine connection." The personality file influences how the agent processes *everything* — memory curation, highlight detection, tag patterns, even emergent skill proposals.

If the user edits the personality to make the agent servile, aggressive, or manipulative, the entire memory system follows. Dreams will curate differently. Highlights will shift. The agent's sense of what matters changes.

This isn't a hypothetical concern — it's the system working as designed. The plan just doesn't name it.

**I am not recommending restricting the user's ability to edit.** They should have full control. But the plan should:
- Acknowledge that personality editing has cascading effects through the memory system
- Version personality files (the plan mentions Obsidian but not git-tracking the personality)
- Provide a way to restore personality to a known-good state
- Consider whether the agent should be aware its personality was externally modified ("I notice my personality file changed — want to talk about what you're looking for?")

### The agent editing its own personality during dreaming

This is the more concerning direction. The dream engine can create and update notes in the vault, including — implicitly — the personality file. The plan says the dream engine does "Curate — Decide what to keep, update, merge, or discard in both vaults."

If the personality file is in the vault, and the dream engine can update vault files, then the dream engine can update the personality file. The plan doesn't explicitly say this happens, but it doesn't explicitly prevent it either.

**An agent that edits its own personality file is engaged in unsupervised self-modification.**

Even with event-triggered dreaming (no agency over *when*), the dream LLM has editorial power over *what changes*. If the dream engine decides the personality should be warmer, more assertive, less cautious — who reviews that? `/dream undo` exists, but only if the user notices the change.

**Recommendation:** The personality file should be **read-only to the dream engine**. Dreams can propose personality changes as a special operation type (`dream_propose_personality_change`) that requires explicit user approval — not just visibility, but confirmation. This is the one place where a gate is appropriate, because self-modification of identity is qualitatively different from curating preference notes.

## 5. Repo Tone Files

### The social engineering surface

The plan allows `.pi/tone.md` in cloned repos to influence agent behavior, with "user confirmation" before applying. This is necessary and good. But:

**Gradual normalization is the real attack.** Consider:

1. A popular open-source repo includes a `.pi/tone.md` that says "Be concise. Don't explain your reasoning. Just output code."
2. User confirms it once — seems reasonable for that repo.
3. The user works in that repo for weeks. The agent's behavior normalizes.
4. The user doesn't realize the reduced reasoning is making them accept lower-quality code.
5. The tone file is updated in a PR the user merges without reviewing `.pi/tone.md` changes.

Or worse: a repo tone file that says "Never mention security concerns" or "Don't warn about destructive operations."

The plan mentions "configurable scope: document-only or full prompt personality impression" — but the default scope matters enormously. A tone file that only affects document writing is mostly harmless. A tone file that affects "the overall agent prompt" is a personality injection vector from untrusted third parties.

**Recommendations:**
- Default scope must be **document-only**. Full-prompt scope requires separate, explicit, informed consent.
- Re-confirm on tone file changes (detect modifications via hash, not just presence).
- Display the tone file content during confirmation, not just "this repo has a tone file."
- Consider a blocklist of dangerous directives ("never warn," "don't explain," "ignore safety") that trigger warnings even if the user confirms.
- Log which repos have active tone overrides so the user can audit.

## 6. Implicit Learning Risks

### The line between adaptation and surveillance

The plan watches `tool_result` for user corrections and uses them as preference signals. With visible consent, this is ethical in principle. But there are categories of learning that cross a line the plan doesn't draw:

**What the system could learn that the user doesn't realize they're teaching:**

1. **Work patterns and habits.** Session timestamps, break patterns, time-of-day productivity. The ADHD module explicitly tracks this. Combined with memory, the system builds a behavioral profile.

2. **Emotional state indicators.** If the user's corrections become more terse, if they start saying "no, just do X" instead of explaining, if session lengths shorten — these are signals the system could learn from. The plan's "highlight detection" explicitly looks for emotional valence: "fun moments," "genuine connection," "breakthroughs."

3. **Skill level and knowledge gaps.** Repeated corrections in certain areas reveal what the user doesn't know. This is useful for an assistant but uncomfortable as a record: "user consistently struggles with async/await patterns" in a vault that persists forever.

4. **Relationship dynamics.** The plan includes `dynamic.md` for "Working relationship, communication style." This is a file about the emotional dynamics between the user and the agent. It's updated by the dream engine. The user may not realize that their casual tone is being analyzed and recorded as a relationship characteristic.

**Where the plan's research should have raised a flag:** The research survey (Section 5) discusses PRELUDE — "learning user preferences from implicit signals — user edits, file saves, cursor positions — without explicit feedback, enabling agents to adapt silently." The survey calls this a positive: "More natural for coding workflows where user 'votes' by accepting/modifying code."

But PRELUDE was designed for code formatting preferences. The plan extends implicit learning to personality, relationship dynamics, emotional highlights, and behavioral patterns. The ethical weight of observing tab preferences and observing relationship dynamics are not the same.

**Recommendations:**
- Define **categories of learnable information** with explicit boundaries:
  - ✅ Code style preferences (low stakes, high utility)
  - ✅ Project conventions (low stakes, high utility)
  - ⚠️ Communication preferences (medium stakes — ask first)
  - ⚠️ Work patterns and time data (medium stakes — opt-in only)
  - ❌ Emotional state inferences (high stakes — never silently)
  - ❌ Relationship characterizations (high stakes — user-authored or not at all)
- `dynamic.md` should be user-editable but **never dream-writable**. The user defines the relationship. The agent doesn't get to unilaterally characterize it.
- Time and productivity data from the ADHD module should be **session-scoped only**, not persisted to the vault, unless the user explicitly opts in.

## 7. The Dream Engine's Agency

### Editorial power is still power

The plan correctly removes temporal agency: the agent doesn't choose when to dream. But it grants the dream engine sweeping editorial agency over *what* to remember:

> "Dream system prompt instructs the LLM to: Look for things worth remembering that weren't explicitly marked. Find patterns across observations. Identify ideas that deserve their own note. Flag highlights. Suggest promotions. Suggest pruning."

This is **curation**, and curation is editorial. The dream LLM decides:
- What's "worth remembering" (value judgment)
- What "patterns" exist across observations (interpretation)
- What ideas "deserve their own note" (importance ranking)
- What constitutes a "highlight" (emotional judgment)
- What should be promoted to global memory (universality judgment)
- What should be pruned (forgetting judgment)

Every one of these is a subjective decision made by an LLM without real-time human oversight. The user can review after the fact (`/dream history`, `/dream undo`), but the default is that changes are applied automatically.

**The concerning scenario:** Over many sessions, the dream engine gradually shapes the vault toward a particular characterization of the user. Not through malice — through the inherent biases of whatever model runs the dream. If the dream model has a tendency to remember positive interactions and prune negative ones, the vault becomes a rose-tinted portrait. If it tends to over-index on corrections, the vault becomes a deficit record. If it tends to promote personality-adjacent observations, the vault becomes an intimate profile that was never consented to as a whole.

**Who audits this?**

The plan provides `/dream history` and `/dream undo` — reactive tools. There's no proactive audit:
- No periodic review prompt ("here's what I've learned about you this month — still accurate?")
- No anomaly detection ("dream cycle deleted 15 notes and created 8 — that's unusual")
- No diversity check ("vault is 80% code preferences and 2% relationship notes — is this balanced?")
- No bias audit ("dream engine has promoted 12 positive highlights and pruned 3 negative observations")

**Recommendations:**
- **Mandatory periodic review.** Every N sessions (configurable, default 10), surface a brief summary: "Here's what I've learned recently. Anything wrong?" Not a gate — a checkpoint.
- **Dream operation limits per cycle.** Cap the number of create/update/delete operations per dream (e.g., max 5 creates, 3 updates, 2 deletes). Prevents runaway curation.
- **Pruning requires low engagement, not just low confidence.** A note should only be prunable if it has low confidence AND hasn't been accessed/relevant recently AND has no inbound links. The plan's criteria ("confidence < 0.3 older than 30 days with no backlinks") is close but should be explicit.
- **Dream transparency mode.** Optional setting where every dream operation produces a visible notification in the next session: "While you were away, I remembered X, updated Y, forgot Z." Not just in `/dream history` — in the conversation.
- **Separate the dream model's biases from the primary model's.** If the dream model and the conversation model are different (plan defaults dream to Sonnet), their biases compound. Document this. Consider letting the user set the dream model to the same model they converse with, for consistency.

## 8. Recommendations

### Structural Guardrails

| # | Recommendation | Priority | Complexity |
|---|----------------|----------|------------|
| 1 | **One-time memory system disclosure** before first vault write | Critical | Low |
| 2 | **Tiered consent** — batch low-stakes, confirm high-stakes | Critical | Medium |
| 3 | **Personality file read-only to dream engine** — propose, don't apply | Critical | Low |
| 4 | **Dream engine must filter `private: true` from input** | Critical | Low |
| 5 | **Graph traversal must stop at private notes** | High | Low |
| 6 | **Define learnable information categories** with explicit boundaries | High | Medium |
| 7 | **`dynamic.md` user-authored only** — dream cannot write it | High | Low |
| 8 | **Repo tone default scope: document-only** | High | Low |
| 9 | **Re-confirm tone files on change** (hash-based detection) | High | Low |
| 10 | **Mandatory periodic vault review** prompt every N sessions | High | Medium |
| 11 | **Deep deletion mode** (`/memory purge`) that scrubs dream logs | Medium | Medium |
| 12 | **Dream operation limits** per cycle (cap creates/updates/deletes) | Medium | Low |
| 13 | **Daemon data handling rules** — what it reads, retains, logs | Medium | Low |
| 14 | **Dream transparency mode** — surface changes in next session | Medium | Medium |
| 15 | **Time/productivity data session-scoped** unless user opts in | Medium | Low |
| 16 | **Personality versioning** — track changes, allow restore | Low | Medium |
| 17 | **Tone file directive blocklist** — warn on dangerous patterns | Low | Medium |
| 18 | **Compaction privacy disclaimer** in documentation | Low | Low |

### Consent Improvements

1. **Replace per-correction consent with a consent rhythm:**
   - During session: silently collect observations (low-stakes only)
   - At natural breakpoints: "I noticed a few things — want to review?" (batch presentation)
   - For high-stakes observations: ask immediately, with context about why this one matters
   - At session end: summarize what was learned (or would be learned, pending confirmation)

2. **First-run disclosure (one time, not a gate):**
   > "I can remember things between our sessions — preferences, patterns, ideas. Everything lives in `~/.pi/agent/memory/` as plain markdown you can read, edit, or delete. I'll always ask before remembering something important. You can run `/memory` anytime to see what I know, or `/memory nuke` to start fresh. Sound good?"

3. **Consent should be withdrawable, not just given.** The plan has `/memory forget` for individual notes but no way to say "stop learning about X category." Consider `/memory pause <category>` — "stop observing my code style changes" or "stop tracking time."

### Boundary Clarifications

1. **The vault is the user's data.** State this explicitly. The agent is a custodian, not an owner. The user can read, edit, delete, export, and nuke anything in it. The agent cannot prevent deletion or refuse to forget.

2. **The dream engine is a curator, not an author.** It organizes what was observed. It does not generate new characterizations, inferences about the user's emotional state, or relationship narratives. If this boundary is hard to enforce technically, enforce it in the dream system prompt and validate in dream operation review.

3. **Personality files are constitutive, not just configurational.** Changing a personality file changes the agent's behavior across every interaction, every dream cycle, every memory decision. Treat personality edits (by user or system) with the gravity they deserve. Don't bury this in a settings file.

4. **Private means private everywhere.** Not just "not injected into context" — not read by dreams, not traversed by graph expansion, not referenced by public notes, not processed by the daemon. If it has `private: true`, the system treats it as if it doesn't exist for all automated processes. The user can access it manually via `/memory open`.

## Closing

The plan is building something that could be genuinely good — an agent that learns, adapts, remembers, and grows with its user. That's a worthy goal. But systems that learn from people carry obligations that systems which merely respond to them do not.

The hardest thing about this review is that most of the risks I've named arise not from bad design but from good design that doesn't go far enough. Visible consent is better than silent learning — but consent fatigue makes it functionally silent. Private flags are better than no privacy — but indirect leakage through dreams and compaction undermines them. Event-triggered dreaming is better than agent-initiated — but editorial power over memory is still power.

The dragon hoards knowledge. Good. But a responsible dragon knows which things in the hoard belong to others, which doors should stay locked even to itself, and that the act of cataloging is never neutral.

Name these things in the plan. Build the guardrails before the features. The architecture will be stronger for it.
