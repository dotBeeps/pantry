# Quest Tool — Implementation Design

> Generated by a wise-griffin-researcher dispatch (10.0 pts)
> Source: Multi-agent orchestration research + pi subagent internals analysis
> Date: 2026-04-07

## Summary

The `quest` tool replaces pi's built-in subagent dispatch with taxonomy-aware orchestration.
Three modes: single quest, rally (parallel), chain (sequential with `{previous}`).
FrugalGPT-style model cascading within each noun tier.
Formula-based budgets with atomic reservation for parallel races.

## Architecture

```
quest tool_call
  → resolveAlly(combo)        // taxonomy → model + tools + prompt
  → budgetReserve(cost)       // atomic pre-deduction
  → spawnPi(args)             // pi --mode json child process
  → streamResults(proc)       // NDJSON → progress updates
  → cascadeOnFailure(chain)   // retry next model on provider error
  → budgetSettle(actual)      // refund delta on complete/fail
  → return result + cost      // structured details for rendering
```

## Key Design Decisions

1. **Three dispatch modes** — quest (single), rally (parallel), chain (sequential). Top-level fields, not mode enum.
2. **Cascade fallback** — ordered model list per tier, cooldown on failure (60s rate limit, 30s server, 5min auth).
3. **Atomic budget reservation** — rally reserves ALL costs before spawning (safe: sync code, single-threaded Node).
4. **Chain reserves step-by-step** — later steps may never run.
5. **Names baked into system prompt file** — no pendingNames bridge needed since quest owns spawn lifecycle.
6. **Both budget cost and provider cost reported** — abstract formula pts AND actual $.
